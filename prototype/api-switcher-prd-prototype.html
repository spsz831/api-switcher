一、需求背景

1. 解决什么问题？

当前开发者在切换 Claude Code、Codex、Gemini CLI 等工具的 API 配置时，常见问题包括：

  • 多个工具配置文件分散在不同目录，格式各不相同，切换成本高。
  • 配置文件里有些字段“看起来存在”，但切换时并不会真正生效，缺乏可解释性。
  • 用户需要手动判断 URL 是否应该补 /api、/v1 或其他平台路径，极易出错。
  • 切换会覆盖本地活动配置，但缺少预览、风险提示和回滚能力，操作心理负担大。
  • 现有工具对新平台扩展能力有限，难以支撑长期演化为公开、稳定、可信赖的通用工具。

2. 覆盖多少用户？

目标用户为需要频繁切换第三方 API 中转站、自建网关或多账号环境的开发者，主要包括：

  • 个人开发者：同时使用多个 CLI 工具进行编码、测试与原型开发。
  • AI 工具重度用户：拥有多个 API 提供方、多个账号与不同访问策略。
  • 团队技术负责人：希望把可复用的配置切换工具公开给团队或社区使用。

3. 上线计划？

Phase 1｜MVP
完成 Claude Code / Codex / Gemini CLI 三平台适配，支持统一配置存储、添加、列出、切换、查看当前配置、风险预检查。
Phase 2｜增强版
加入配置验证、切换预览、自动备份、回滚、导入导出、敏感信息脱敏展示。
Phase 3｜公开发布
完善 README、命令帮助、示例模板、测试覆盖与 npm 发布流程，形成稳定的公开工具。

4. 还需要准备什么？

  • 梳理三类平台的配置文件路径、活动配置格式与当前可验证写入策略。
  • 设计统一配置 schema，明确“存档字段”与“切换生效字段”边界。
  • 定义平台适配器接口，确保未来新增平台不影响核心命令层。
  • 建立本地回滚机制，避免切换失败导致环境不可用。

三、细节方案

3.1 统一配置中心

采用 source / apply 双层模型，解决旧工具“字段存在但切换不生效”的问题。

{
  "name": "wong-prod",
  "platform": "claude",
  "source": {
    "baseURL": "https://wzw.pp.ua/api",
    "token": "sk-***",
    "notes": "主力节点"
  },
  "apply": {
    "ANTHROPIC_AUTH_TOKEN": "sk-live-***",
    "ANTHROPIC_BASE_URL": "https://wzw.pp.ua/api"
  }
}

3.2 平台适配器系统

  • ClaudeAdapter：settings.json 映射与预览
  • CodexAdapter：config.toml / auth.json 生成
  • GeminiAdapter：Gemini CLI 配置适配与未来扩展

五、评估与风险

风险 01
平台配置规范变化

不同 CLI 官方配置格式可能更新，需要适配器独立维护与测试用例覆盖。

风险 02
用户配置来源复杂

用户可能手改过现有活动文件，导致 current 检测不稳定，需要更健壮的比对策略。

风险 03
安全感知不足

如果工具输出过多敏感信息或导出机制过于随意，会削弱公开工具的可信度。

风险 04
功能膨胀

如果过早加入同步、云端账户体系等高级能力，可能让 MVP 失焦，应优先打磨核心切换体验。

Part 04

痛点解决方案详解

痛点一：切换生效范围不透明

通过 preview 页面直接展示会真正写出的字段与只是存档字段，让用户在切换前形成准确预期。

痛点二：URL 路径容易猜错

加入路径风险提示与平台建议，明确告诉用户当前 URL 看起来缺少什么，而不是在切换失败后才暴露问题。

痛点三：覆盖配置没有安全感

默认先备份再写入，并在结果中展示回滚点，帮助用户建立可逆操作的信任。
