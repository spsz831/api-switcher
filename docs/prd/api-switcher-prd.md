# 一、需求背景

## 1. 解决什么问题？

api-switcher 要解决的是开发者在多平台 CLI 工具之间切换 API 配置时的高认知成本与高出错率问题。

当前用户在使用 Claude Code、Codex、Gemini CLI 等工具时，往往会遇到以下问题：

1. **配置文件分散**：不同工具的配置文件路径不同、格式不同、命名不同，难以统一管理。
2. **字段语义不透明**：配置文件中有些字段只是“存档”，并不会在切换时真正生效，但用户无法直观看出来。
3. **平台规则不一致**：不同平台对 URL、认证字段、目标文件的要求不同，用户需要自己记忆或猜测。
4. **手工切换风险高**：直接修改本地活动配置容易覆盖原状态，一旦写错，恢复成本高。
5. **现有工具扩展性有限**：如果未来要加入 Gemini CLI 或更多平台，旧式硬编码方案会迅速失去可维护性。

因此，api-switcher 的核心价值不是“帮用户改文件”，而是：

- 提供统一配置中心
- 提供平台适配器机制
- 提供切换前可解释的风险预览
- 提供切换后可回滚的安全保障
- 提供长期可扩展的公开工具架构

## 2. 覆盖多少用户？

目标用户主要为以下几类：

### 2.1 个人开发者
- 同时使用多个 AI CLI 工具进行编码、调试、原型开发
- 拥有多个 API 提供方或中转服务
- 需要频繁切换账号、节点、环境

### 2.2 AI 工具重度用户
- 同时使用 Claude、Codex、Gemini 等平台
- 希望统一管理多个 API key、base URL、环境用途
- 对切换速度、可解释性和稳定性要求高

### 2.3 技术团队与开源维护者
- 希望为团队建立统一可复用的 API 配置切换方式
- 希望将工具开源并持续迭代
- 重视架构清晰度、可扩展性与安全边界

### 2.4 市场规模判断
这类产品不是大众消费型产品，而是面向开发者的专业工具。其价值不在用户规模的绝对值，而在于：

- 用户使用频次高
- 对体验与稳定性敏感
- 容易形成口碑传播
- 一旦做对，会成为高粘性的长期工具

## 3. 上线计划？

### Phase 1：MVP
目标：建立最小可用版本，完成核心闭环。

范围：
- 支持 Claude Code / Codex / Gemini CLI 三平台
- 支持统一配置存储
- 支持 add / list / use / current
- 支持 preview / validate
- 支持基础备份与回滚

### Phase 2：增强版
目标：提升稳定性与公开可用性。

范围：
- 更完善的字段校验
- 更精细的差异预览
- 导入导出
- 脱敏显示
- 更完整的错误提示和恢复机制

### Phase 3：公开发布版
目标：形成稳定、可信、可扩展的公开工具。

范围：
- README 与示例完善
- 单元测试与集成测试覆盖
- npm 发布流程
- 平台适配文档
- 版本演进与 CHANGELOG 规范

## 4. 还需要准备什么？

1. 梳理三平台当前活动配置写入位置和文件格式。
2. 定义统一配置 schema。
3. 明确 source / apply 双层结构。
4. 设计平台适配器接口。
5. 设计 preview、validate、rollback 的行为规范。
6. 确定本地备份策略。
7. 设计日志、脱敏与危险操作确认策略。

## 6. MVP 范围

MVP 必须聚焦“把最核心的切换闭环做对”，避免过早膨胀。

### 6.1 MVP 必做
- 支持 Claude Code / Codex / Gemini CLI 三平台
- 支持统一配置中心读写
- 支持 `add`、`list`、`use`、`current`
- 支持 `preview` 与 `validate`
- 支持自动备份最近一次活动配置
- 支持 `rollback` 恢复最近快照
- 支持脱敏输出
- 支持最基础的平台 URL 风险提示

### 6.2 MVP 可选但尽量争取
- 支持交互式添加配置
- 支持 `--json` 输出，方便后续脚本化调用
- 支持按平台筛选 `list`
- 支持 `use --dry-run` 直接走预览逻辑

## 7. 非 MVP 范围

以下能力很有价值，但不应阻塞首发：
- WebDAV / 云同步
- 团队共享配置
- GUI 桌面端
- 远程配置仓库
- 自动测速与节点健康检查
- 自动轮询可用性
- 登录态管理
- 配置市场 / 模板市场

## 8. 成功标准

如果 MVP 达到以下状态，可以视为具备首发价值：
- 用户能在 10 分钟内完成首次添加与切换
- 用户能清楚理解哪些字段会真正生效
- 切换失败时可以恢复到之前状态
- Claude / Codex / Gemini 三平台至少各有一条通过验证的可用切换链路
- README 能独立支撑新用户完成上手

# 二、方案概述

## 2.1 核心业务流程

用户先在统一配置中心中维护多平台配置档。每条配置档包含完整存档信息与切换时真正生效的字段映射。用户在执行切换前，可以通过 preview 查看：

- 将写入哪些目标文件
- 哪些字段会真正生效
- 哪些字段只是存档
- 当前配置是否存在风险提示
- 是否会创建回滚快照

确认后，系统通过对应的平台适配器执行写入，并在成功后输出结果与回滚点。

## 2.2 核心功能流程示意

1. 用户选择配置档
2. 系统读取统一配置文件
3. 根据 platform 调用对应 adapter
4. adapter 执行 validate 与 preview
5. 系统生成备份
6. adapter 将 apply 字段写入目标活动配置文件
7. 成功后输出 current 状态与 rollback snapshot

## 2.3 典型用户旅程

### 2.3.1 首次上手旅程
1. 用户安装 api-switcher。
2. 用户执行 `add` 新增一条 Claude / Codex / Gemini 配置。
3. 系统立即返回 validate 结果与 preview 摘要。
4. 用户执行 `use` 完成首次切换。
5. 用户执行 `current` 确认活动配置，并看到最近回滚点。

这条旅程的目标是让新用户在最短路径内建立对工具的信任。

### 2.3.2 高频切换旅程
1. 用户已有多条不同用途的 profile，例如 `prod`、`backup`、`office`。
2. 用户通过 `list` 查看当前状态与风险标签。
3. 用户执行 `preview <profile>` 或 `use <profile> --dry-run` 先确认影响面。
4. 用户执行正式切换。
5. 如果结果不符合预期，用户通过 `rollback` 恢复最近快照。

这条旅程的目标是把“频繁切换”从危险操作变成低压力日常操作。

### 2.3.3 故障恢复旅程
1. 用户切换后发现目标 CLI 无法正常工作。
2. 用户执行 `current` 或 `rollback list` 查看最近快照。
3. 用户先预览要恢复的文件，再执行回滚。
4. 系统恢复目标文件并更新 state。
5. 用户重新验证活动配置是否恢复正常。

这条旅程的重点不是“切成功”，而是“切错也能安全回来”。

## 2.4 产品体验原则

api-switcher 的体验必须围绕以下原则设计：

- **先解释，再写入**：默认先 validate / preview，再进入 apply。
- **默认安全**：默认脱敏、默认备份、默认保留恢复路径。
- **跨平台一致**：不同平台底层行为不同，但命令体验要尽量一致。
- **结果可追踪**：用户要知道当前是谁生效、刚刚改了什么、回滚点在哪里。
- **脚本友好**：交互体验不能妨碍后续自动化集成。

# 三、细节方案

## 3.1 统一配置中心

新工具不直接沿用旧项目“把所有字段混在一起”的方式，而是采用统一配置中心。

### 3.1.0 设计目标

统一配置中心必须同时满足三件事：
- 对用户来说好理解：配置名、平台、用途、风险一眼能看懂
- 对程序来说好处理：字段稳定、结构明确、适配器易于消费
- 对未来来说好扩展：新增平台时尽量不破坏已有 schema

建议将统一配置文件作为“唯一配置源”，各平台当前活动文件都由它派生生成。

默认建议文件命名可暂定为：
- `profiles.json`：用户维护的统一配置源
- `backups/`：回滚快照目录
- `state.json`：记录当前活动配置、最近切换结果、快照索引等运行态信息

> 这里的命名仍可在技术设计阶段再细化，但 PRD 层面需要先明确“唯一配置源 + 运行态分离”的方向。


统一配置中心负责：
- 存放所有平台配置档
- 维护配置 identity 信息
- 保存原始 source 数据
- 保存切换时真正生效的 apply 数据
- 记录元数据与风险信息

### 3.1.1 source / apply 双层模型

这是 api-switcher 的核心升级。

#### source
表示配置存档层，用于保存：
- 原始 token / key
- base URL
- 用户备注
- 标签
- 其他说明字段

#### apply
表示切换执行层，用于保存：
- 目标平台在切换时真正要写入的字段
- 明确哪些值会写入到当前活动配置文件

通过这种分层：
- 用户不再误以为“所有字段都会生效”
- preview 可以明确展示哪些字段真正起作用
- 不同平台可以有自己的字段映射逻辑

### 3.1.2 配置 schema 设计原则

建议 schema 结构如下：

```json
{
  "version": 1,
  "profiles": [
    {
      "id": "claude-wong-prod",
      "name": "wong-prod",
      "platform": "claude",
      "source": {
        "baseURL": "https://wzw.pp.ua/api",
        "token": "sk-live-xxx",
        "notes": "主力节点"
      },
      "apply": {
        "ANTHROPIC_AUTH_TOKEN": "sk-live-xxx",
        "ANTHROPIC_BASE_URL": "https://wzw.pp.ua/api"
      },
      "meta": {
        "tags": ["prod", "stable"],
        "riskLevel": "low",
        "createdAt": "2026-04-04T12:00:00+08:00",
        "updatedAt": "2026-04-04T12:00:00+08:00"
      }
    }
  ]
}
```

#### schema v1 字段说明

| 字段 | 是否必需 | 说明 |
|:--|:--|:--|
| version | 是 | 配置文件版本，便于未来迁移 |
| profiles | 是 | 配置档数组 |
| profiles[].id | 是 | 稳定唯一 ID，用于内部引用和回滚记录 |
| profiles[].name | 是 | 面向用户展示的配置名称 |
| profiles[].platform | 是 | 平台标识，首批支持 `claude`、`codex`、`gemini` |
| profiles[].source | 是 | 原始存档信息 |
| profiles[].apply | 是 | 切换时实际写入的字段 |
| profiles[].meta | 否 | 标签、风险等级、创建时间等扩展信息 |

#### schema v1 约束
- `id` 必须全局唯一
- `name` 允许重复但不推荐，CLI 默认优先通过 `id` 精确定位
- `platform` 必须对应已注册 adapter
- `apply` 必须通过该平台 adapter 的 `validate` 校验
- `source` 可比 `apply` 更丰富，但不得与 `apply` 的核心语义冲突

原则：
- identity、source、apply、meta 语义清晰
- 支持未来扩展更多平台
- 支持 preview 与 validate 的可解释输出
- 不让平台内部逻辑污染用户存档结构
- 允许通过配置版本做平滑升级和迁移

### 3.1.3 state 与 backup 运行态设计

除了统一配置源，建议额外维护运行态文件，用于减少对用户配置文件的污染。

建议结构：

```json
{
  "current": {
    "claude": "claude-wong-prod",
    "codex": "codex-ice-backup",
    "gemini": "gemini-office"
  },
  "lastSwitch": {
    "platform": "claude",
    "profileId": "claude-wong-prod",
    "time": "2026-04-04T12:48:06+08:00",
    "backupId": "snapshot-20260404-124806"
  }
}
```

这类运行态信息不应混入 `profiles.json` 中，以避免用户配置与系统状态耦合。

### 3.1.4 配置迁移策略

考虑到后续 schema 可能继续演进，产品一开始就应该支持迁移机制。

建议：
- 配置文件头部保留 `version`
- 程序启动时先执行迁移器
- 老版本结构通过 `migrations/` 脚本升级到当前版本
- 对无法自动迁移的情况输出明确提示并中止写入

这样可以保证公开工具未来升级时不轻易破坏用户已有配置。

### 3.1.5 脱敏展示规则

所有输出层默认必须脱敏。

建议规则：
- API key / token：仅显示前 3~6 位和后 2~4 位
- URL：完整展示，但可对 query 参数脱敏
- preview：默认展示脱敏值，仅在明确指定 `--show-secrets` 时展示完整值
- export：默认导出脱敏版本，原始导出需显式确认

这样可以降低截图、日志和演示时的泄露风险。

### 3.1.6 配置健康度与风险标签

在 `meta` 中建议支持以下可选字段：
- `riskLevel`: `low | medium | high`
- `healthStatus`: `unknown | valid | warning | invalid`
- `lastValidatedAt`
- `notes`

这些字段不直接参与切换，但会显著提升列表展示、预览决策和未来的可视化管理体验。

## 3.2 平台适配器系统

平台适配器是整个架构的核心扩展机制。

每个平台都必须实现统一接口，例如：
- validate(config)
- preview(config)
- backup()
- apply(config)
- detectCurrent()
- listTargets()
- rollback(snapshot)

### 3.2.0 平台接口契约

建议定义一个统一的 adapter 契约：

```ts
interface PlatformAdapter {
  readonly platform: string
  validate(profile: Profile): ValidationResult
  preview(profile: Profile): PreviewResult
  detectCurrent(): CurrentProfileResult | null
  backup(): BackupResult
  apply(profile: Profile): ApplyResult
  rollback(snapshotId: string): RollbackResult
  listTargets(): TargetFileInfo[]
}
```

#### 统一返回值原则
- 所有 adapter 返回结构尽量一致
- 不直接向命令层暴露平台内部细节
- 命令层只关心“是否成功、影响了哪些文件、有哪些提醒、能否回滚”

#### PreviewResult 建议字段
- `platform`
- `targetFiles`
- `effectiveFields`
- `storedOnlyFields`
- `warnings`
- `requiresConfirmation`

#### ValidationResult 建议字段
- `ok`
- `errors`
- `warnings`
- `normalizedSuggestions`

这样能让 CLI 输出与未来 UI 展示都保持统一。

### 3.2.1 ClaudeAdapter

负责：
- 读取和写入 Claude Code 的活动配置文件
- 识别当前配置是否匹配某个档案
- 对 source/apply 进行字段映射
- 提示 URL 是否可能缺少 /api
- 在 preview 中清楚区分“有效字段”与“存档字段”
- 尽量保留非本工具管理字段，避免粗暴覆盖

#### ClaudeAdapter 最低能力要求
- 能列出目标写入文件
- 能生成 apply 与当前活动配置的差异
- 能识别现有 settings 中的非托管字段
- 能生成可恢复快照

### 3.2.2 CodexAdapter

负责：
- 生成并写入 Codex 对应活动配置
- 处理 config.toml 与 auth.json 的写入
- 提示 URL 是否更适合 /v1 或 /openai/v1
- 对 key 与 URL 进行平台特定校验
- 在 preview 中展示多文件写入结果

#### CodexAdapter 最低能力要求
- 同时支持两个目标文件的写入预览
- 能把统一配置映射成最终 config/auth 内容
- 能在回滚时同时恢复多个文件

### 3.2.3 GeminiAdapter

负责：
- 抽象 Gemini CLI 的配置写入行为
- 建立统一 adapter 接口
- 支持未来根据 Gemini CLI 真实配置格式做适配升级
- 把 Gemini 相关风险提示纳入统一 validate/preview 流程

设计原则：
- Gemini 不是例外，而是与 Claude / Codex 同级的标准平台适配器
- 接入新平台不应修改命令层逻辑，只需新增 adapter
- 如果 Gemini 配置规范变化，影响应局限在 GeminiAdapter 内部

### 3.2.4 新平台接入标准

未来如果要增加更多平台，建议要求其至少满足：
- 有明确的活动配置文件目标
- 能识别当前配置状态
- 能提供有效字段与仅存档字段区分
- 能做预检、写入、回滚

如果某个平台做不到回滚，也必须明确在 preview 中标示这一限制。

### 3.2.5 平台能力矩阵

| 能力 | Claude | Codex | Gemini |
|:--|:--|:--|:--|
| 单文件写入 | 支持 | 部分场景 | 待适配确认 |
| 多文件写入 | 低概率 | 高概率 | 待适配确认 |
| source / apply 映射 | 必需 | 必需 | 必需 |
| current 检测 | 必需 | 必需 | 必需 |
| 自动备份 | 必需 | 必需 | 必需 |
| 回滚支持 | 必需 | 必需 | 目标支持 |
| URL 风险提示 | `/api` 类提示 | `/v1` / `/openai/v1` 类提示 | Gemini 网关兼容提示 |

说明：
- 这里的能力矩阵不是实现细节，而是 PRD 层面的最低产品承诺。
- 对于 Gemini 等后续仍需调研的部分，应在 TDD 中继续落地为真实文件契约。

## 3.3 命令层设计

建议的核心命令包括：

| 命令 | 作用 |
|:--|:--|
| list | 查看配置档列表 |
| add | 新增配置档 |
| use | 切换到指定配置档 |
| current | 查看当前活动配置 |
| preview | 预览切换影响面 |
| validate | 检查配置合法性与风险 |
| rollback | 回滚到指定快照 |
| export | 导出配置 |

命令层设计原则：
- 常用命令直观简短
- 高风险命令要强调确认与说明
- 输出信息必须面向理解，而不只是面向执行
- 所有命令尽量支持非交互模式，方便脚本调用

### 3.3.1 命令详细定义

#### list
作用：列出配置档

建议参数：
- `--platform <platform>`
- `--json`
- `--all`

输出重点：
- 名称
- 平台
- 当前状态
- 风险标签
- 最近验证状态

#### add
作用：新增配置档

建议参数：
- `--platform`
- `--name`
- `--url`
- `--key`
- `--interactive`

设计要求：
- 支持交互式添加
- 支持直接命令式添加
- 添加后立即提示 preview/validate 结果

#### use
作用：切换配置

建议参数：
- `<profile-id-or-name>`
- `--dry-run`
- `--force`

设计要求：
- 默认先做 validate 与 preview
- 如果存在高风险提示，要求明确确认
- 成功后输出回滚点

#### current
作用：查看当前活动配置

输出重点：
- 当前平台
- 当前配置档
- 活动文件位置
- 最近切换时间
- 可回滚点

#### preview
作用：预览切换影响面

输出重点：
- 将写入的目标文件
- effective fields
- stored-only fields
- warnings
- current 与 target 差异摘要

#### validate
作用：校验配置

支持：
- 校验单条配置
- 校验全部配置

#### rollback
作用：恢复快照

支持：
- 列出可用快照
- 恢复指定快照
- 恢复前再次展示影响文件

#### export
作用：导出配置

支持：
- 脱敏导出
- 原始导出（高风险确认）
- 按平台导出

### 3.3.2 命令输出设计原则

命令输出应分层：
- 标题层：当前在做什么
- 结果层：成功/失败/警告
- 解释层：为什么这样、接下来可以做什么

避免只输出“Done”或“Failed”这类低信息密度文本。

### 3.3.3 退出码策略

为了兼容自动化脚本，建议定义：
- `0`：成功
- `1`：业务失败（校验失败、配置不存在等）
- `2`：运行异常（I/O、解析失败等）

这样便于未来在 CI 或自动化脚本中集成。

### 3.3.4 命令兼容与命名约束

为了降低认知成本，命令层还应满足以下约束：
- 同一类参数命名尽量统一，例如都使用 `--platform`、`--json`、`--force`
- 高风险行为尽量不依赖隐式默认值
- 支持脚本模式时，输出应尽量稳定，避免频繁变更字段名
- 交互式提示与非交互式输出要分层，不相互污染

这意味着 api-switcher 既要像“好用的 CLI”，也要像“可嵌入的底层工具”。

## 3.4 切换预览与风险校验

在产品层面，preview 和 validate 不是附属命令，而是 use 命令的前置能力。

理想体验应是：
- 用户单独执行 `preview` 时，能独立理解即将发生什么
- 用户执行 `use` 时，即使不手动先跑 preview，也能看到足够清晰的摘要
- 用户在脚本模式下可以关闭冗余说明，但不能失去必要错误信息
- 同一份风险判断逻辑，应被 `preview`、`validate`、`use` 共同复用

这会直接影响后续服务层与 adapter 层的职责划分。

preview 是 api-switcher 的关键能力。

用户在执行 use 前，应能看到：
- 目标平台
- 目标活动文件
- 将写入的 apply 字段
- 仅存档不会写出的字段
- URL 路径建议
- 当前文件是否存在手工编辑痕迹
- 是否建议先创建回滚点

validate 负责：
- 校验 key 是否为空
- 校验 URL 是否格式异常
- 校验平台是否支持
- 校验字段冲突
- 校验 apply 是否缺少必要字段

### 3.4.1 风险等级建议

建议把风险分为三层：
- `low`：正常切换，无明显问题
- `medium`：可切换，但建议确认，例如 URL 可能缺少后缀
- `high`：不建议直接切换，例如目标字段缺失、平台未注册、配置无法解析

### 3.4.2 风险类型建议

可按以下类别输出：
- `url-path-warning`
- `missing-required-field`
- `unmanaged-current-file`
- `multi-file-overwrite`
- `rollback-unavailable`
- `platform-unsupported`

### 3.4.3 Preview 页面/输出的核心价值

preview 不只是“确认框”，而是产品体验的核心。

它帮助用户回答三个问题：
1. 会改哪里？
2. 哪些值真的会生效？
3. 如果结果不对，我能不能退回来？

### 3.4.4 高风险确认策略

对于 `use`、`rollback`、`export --raw` 这类可能影响本地状态或暴露敏感信息的命令，建议采用统一确认策略：

- `low` 风险：默认直接执行，但输出摘要
- `medium` 风险：默认弹出确认提示，非交互模式下要求显式 `--force`
- `high` 风险：默认拒绝执行，除非用户显式确认且不存在结构性错误

确认提示应至少包含：
- 即将写入的目标文件数量
- 风险原因摘要
- 是否已创建备份
- 回滚是否可用

这样可以避免把 `--force` 设计成一个“看不见后果的万能开关”。

## 3.5 备份与回滚

每次切换前都应自动生成备份快照。

快照需记录：
- 平台
- 时间
- 来源配置名
- 目标文件
- 原始活动配置内容

回滚能力应支持：
- 列出可恢复快照
- 查看快照来源信息
- 恢复到某个稳定状态
- 恢复后重新标记 current

### 3.5.1 快照目录建议

建议目录结构：

```text
backups/
  claude/
  codex/
  gemini/
```

单次快照可包含：
- 快照元信息 JSON
- 被备份的原始文件内容
- 恢复目标列表

### 3.5.2 回滚策略

建议优先实现“最近一次快照恢复”，再扩展到“指定快照恢复”。

这样可以先把最常见的失败恢复场景做扎实。

### 3.5.3 快照保留与清理规则

为了避免备份无限增长，建议在 PRD 层面先定义保留原则：

- 默认至少保留每个平台最近 N 次快照
- 最近一次成功切换前的快照不得被自动清理
- 清理动作只删除旧快照，不影响 `profiles.json` 与 `state.json`
- 如果某个快照仍被 `lastSwitch` 或当前回滚索引引用，则不可删除
- 清理前应支持 dry-run 或至少输出将删除哪些快照

该规则的核心目标是：既保住恢复能力，也避免备份目录长期失控。

## 3.6 项目目录结构建议

```text

  constants/
tests/
  unit/
  integration/

```

### 3.6.1 目录职责建议

# 四、非功能性需求

## 4.1 性能需求

- list / current / validate 在普通配置规模下应快速完成
- 单次切换不应有明显卡顿
- 尽可能避免重复写入无变化文件
- 文件比对与备份逻辑应控制在轻量级 I/O 范围内

## 4.2 安全需求

- 默认输出必须脱敏显示 token / key
- 导出支持脱敏模式
- 危险操作需确认
- 回滚快照默认本地保存，不自动上传
- 错误日志不应泄露完整敏感值

## 4.3 兼容性需求

- 支持 Windows、macOS、Linux
- 支持 Node.js LTS
- 兼容不同换行符与文件路径规范
- 适配常见 shell 和本地开发环境

# 五、评估与风险

## 5.1 评估指标

- 首次成功完成配置切换的用户比例
- 因路径错误导致的切换失败率
- 使用 preview / validate 的比例
- 回滚成功率
- 新增平台适配平均工作量
- README 到成功使用的时间

## 5.2 潜在风险

### 风险 1：平台配置规范变化
不同 CLI 的官方配置格式可能变更，需通过 adapter 解耦影响。

### 风险 2：用户本地已有非标准修改
如果用户手工编辑了活动文件，current 检测和差异预览可能变复杂。

### 风险 3：过度功能膨胀
如果太早加入同步、账号体系、云端管理等，会影响 MVP 打磨。

### 风险 4：安全边界模糊
如果日志、导出、异常处理不严格，公开工具的可信度会下降。

- 脱敏分享链接
- 本地 UI 包装层

但这些都应建立在首个 CLI 核心版本稳定之后。

## 3.14 当前结论

在现阶段，api-switcher 的产品重点应非常明确：
- 先把统一配置中心做对
- 先把三平台切换闭环做对
- 先把 preview / validate / rollback 做成核心卖点
- 先把 schema 和 adapter 边界定清楚

只要这几个点做扎实，这个产品就具备长期演进和公开发布的基础。

# 四、非功能性需求

## 4.1 性能需求

- list / current / validate 在普通配置规模下应快速完成
- 单次切换不应有明显卡顿
- 尽可能避免重复写入无变化文件
- 文件比对与备份逻辑应控制在轻量级 I/O 范围内

## 4.2 安全需求

- 默认输出必须脱敏显示 token / key
- 导出支持脱敏模式
- 危险操作需确认
- 回滚快照默认本地保存，不自动上传
- 错误日志不应泄露完整敏感值

## 4.3 兼容性需求

- 支持 Windows、macOS、Linux
- 支持 Node.js LTS
- 兼容不同换行符与文件路径规范
- 适配常见 shell 和本地开发环境

# 五、评估与风险

## 5.1 评估指标

- 首次成功完成配置切换的用户比例
- 因路径错误导致的切换失败率
- 使用 preview / validate 的比例
- 回滚成功率
- 新增平台适配平均工作量
- README 到成功使用的时间

## 5.2 潜在风险

### 风险 1：平台配置规范变化
不同 CLI 的官方配置格式可能变更，需通过 adapter 解耦影响。

### 风险 2：用户本地已有非标准修改
如果用户手工编辑了活动文件，current 检测和差异预览可能变复杂。

### 风险 3：过度功能膨胀
如果太早加入同步、账号体系、云端管理等，会影响 MVP 打磨。

### 风险 4：安全边界模糊
如果日志、导出、异常处理不严格，公开工具的可信度会下降。

# 总结

api-switcher 的目标不是做一个“能跑”的脚本，而是打造一款可长期维护、可公开发布、可持续扩展的多平台 API 配置切换工具。

它的核心升级点包括：
- source / apply 双层配置模型
- 平台适配器架构
- 切换预览与风险校验
- 备份与回滚能力
- 面向公开工具的安全与可维护性设计

这是一个非常适合长期迭代的开发者工具方向。
